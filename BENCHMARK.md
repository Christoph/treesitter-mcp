# BENCHMARK.md

How to evaluate whether `treesitter-mcp` actually improves agent workflows — not just
whether it produces smaller payloads.

The existing token-efficiency tests under
`tests/token_efficiency_test.rs` answer one question: "for a given workflow, how many
tokens does the MCP payload contain vs an agent-style shell baseline?". That is a
**compression** benchmark. It does not prove that an agent armed with this server
finishes tasks correctly or faster. The accuracy benchmark described here does.

## Goals

1. Measure **task success** — did the agent produce a correct edit / answer / patch?
2. Measure **turn count** — how many tool calls did it take?
3. Measure **total tokens** (input + output, including tool responses).
4. Measure **hallucination rate** — how often the agent invents symbols, fields, or
   signatures that do not exist in the repo.
5. Compare two configurations: **with** `treesitter-mcp` and **without** (plain `Read` +
   `Grep` + `Glob`).

## Scope

Three task types, 20 tasks each, 60 total. Keep the corpus small enough to run in a
single afternoon; large enough to detect a double-digit percentage-point effect.

### Task type A — Refactor / rename

Each task specifies a symbol and a new name. Success = the diff renames the symbol and
every usage site, changes nothing else, and the target project still passes its tests.

Sample:

- `RustProject::calculator.rs`: rename `perform_sequence` to `run_sequence`, update all
  call sites.
- `typescript_project/calculator.ts`: rename `applyOperation` to `applyOp`, update
  imports in the same file set.
- `complex_rust_service`: rename the public method `build_report` to
  `build_invoice_report`, keep visibility and comments intact.

### Task type B — Signature change with ripple

Each task specifies a function and a new signature. Success = signature changed at the
definition, every call site updated to match, no stale callers, tests pass.

Sample:

- Change `detect_language(path: impl AsRef<Path>) -> Result<Language>` to accept a
  second `hint: Option<&str>` parameter. Update all callers to pass `None`.
- Change a TypeScript function from returning `string` to returning `{ value: string;
  at: number }`. Update every call site.

### Task type C — Targeted question

The agent answers a question about the repo. Success = answer includes a specific
fact that a grader can check (symbol name, file path, line number, caller count).

Sample:

- "Which function in `src/analysis/` owns the diagnostics formatting path?"
- "How many call sites does `parse_code` have inside `src/analysis`?"
- "Give the full signature of `extract_enhanced_shape` and its return type."

## Corpus construction

- Use repositories already checked in under `tests/fixtures/` plus `src/` itself so the
  tasks have ground truth that can be regenerated from git state.
- For each task, store the ground-truth diff (for A and B) or the ground-truth answer
  (for C) in `benches/accuracy/tasks/<id>.json`.
- Mark tasks that require cross-file awareness so we can slice results.

## Running an agent

The harness should not care which agent runs the task. Two targets are enough to start:

1. **Claude Code CLI** — already the primary user surface for this server.
2. **A minimal scripted agent** — loops tool calls with a small system prompt so the
   benchmark is reproducible even when we are not iterating on real agents.

Each run is isolated in a fresh git checkout of the fixture. The harness records:

- the ordered list of tool calls
- every input and output, with tiktoken byte counts
- the final diff (for A and B) or final answer (for C)
- wall-clock time

## Configurations

Two per agent per task — one pair per row in the results table.

- **baseline**: `Read`, `Grep`, `Glob` only. No MCP server registered.
- **mcp**: add `treesitter-mcp` to the MCP config. Bootstrap prompt includes the short
  tool-selection snippet described in `FUTURE.md` item 13.

Run both configurations with the same agent, same system prompt, same random seed where
applicable.

## Metrics

For every task/configuration:

| Metric | Definition |
|---|---|
| `success` | Ground-truth diff matches (A, B) or checkable fact is correct (C). Boolean. |
| `turns` | Number of tool calls the agent issued. |
| `tokens_in` | Sum of tokens sent to the model (system + user + tool responses). |
| `tokens_out` | Sum of tokens generated by the model. |
| `hallucination` | Count of symbols or paths referenced in model output that do not exist in the repo. Detected by post-processing the transcript against a repo symbol index. |
| `wall_clock_s` | Seconds from start to success / giveup. |

## Reporting

Emit one CSV row per (task, configuration). Summarise in README the **difference**, not
the absolute numbers, to avoid cherry-picking:

- Δ success rate (percentage points)
- Δ turns (median)
- Δ total tokens (median)
- Δ hallucination count (median)

Example target phrasing for the README, once we have numbers:

> Across 60 tasks, agents running with `treesitter-mcp` completed X percentage points
> more tasks successfully, used Y% fewer total tokens, and referenced N% fewer
> nonexistent symbols than the `Read`+`Grep` baseline.

That is a real claim. The current multipliers (5×, 57×) are **compression**, not
**outcome**.

## Hallucination detection

The simple version: after the run finishes, build a project-wide symbol index (tree-sitter
extraction of function, method, struct, class, enum, field, type-alias names) and flag
any backticked identifier or function-call-shaped token in the transcript that is not in
the index and not a standard-library identifier. False positives are tolerable; we care
about the **delta** between configurations.

A more precise version later: run the transcript against a language server to resolve
each reference, then flag unresolved ones.

## Running the harness

Once implemented, the flow should look like this:

```bash
cargo run -p accuracy_bench -- \
  --corpus benches/accuracy/tasks \
  --agent claude-code \
  --config mcp \
  --out benches/accuracy/results/$(date +%s)_mcp.csv
```

A second run with `--config baseline` produces the comparison file.

## Guardrails against gaming the result

- Fix the corpus by commit hash. A PR cannot silently add tasks where MCP shines.
- Fix the agent prompt and model version per benchmark run; record both in the result
  file.
- Keep the task ground truth in a file that predates the run. No post-hoc authoring.
- Publish the CSVs, not only the summary. Reviewers can re-aggregate.

## Status

Not yet implemented. This document describes the design we would build next; it is a
plan, not a result. The existing tests under `tests/token_efficiency_test.rs` remain the
only measured claim in the project today, and they only cover compression.
